## Location
`backends/backend-minigame/src/main/java/com/github/xpenatan/gdx/teavm/backends/minigame/config/backend/MiniGameBackend.java` — `generateGameJs()` around line 174

## The Bug
game.js does:
```javascript
const mainModule = require('./output.js');
if (mainModule && mainModule.main) {
    mainModule.main([...args]);
}
```

But TeaVM's `TeaVMTargetType.JAVASCRIPT` output is an **IIFE** (Immediately Invoked Function Expression) that defines `main()` as a **global function**, NOT a CommonJS module export. So:
- `mainModule` will be `{}` (empty object from require)
- `mainModule.main` will be `undefined`
- **The game will never start**

Compare with WebBackend for JS mode: `mode = "main(%ARGS%)"` — it calls `main()` directly as a global function in the browser.

For WeChat, `require()` wraps the script in a function scope, so globals defined inside may not leak out. This is a fundamental module system mismatch.

## Possible Fixes

### Option A: Use globalThis to bridge
In the adapter, set up a bridge:
```javascript
// adapter/index.js
globalThis.$startGame = function(mainFunc) {
    mainFunc([...args]);
};
```
Then at the end of the TeaVM output, inject: `if(typeof $startGame==='function') $startGame(main);`

### Option B: Don't use require() — use eval or fs.readFile
```javascript
const fs = wx.getFileSystemManager();
const code = fs.readFileSync('./output.js', 'utf-8');
// eval would execute in global scope
```

### Option C: Wrap the TeaVM output to export main
Post-process the output.js to wrap it in a module.exports pattern, or inject `module.exports = { main: main };` at the end.

### Option D: Use TeaVM's WasmGC target instead
The WasmGC target uses `teavm.exports.main()` which is an explicit export. MiniGameBackend already has `targetType = TeaVMTargetType.JAVASCRIPT` but could be changed.

**This is likely the #1 reason the game won't start in WeChat DevTools.**
