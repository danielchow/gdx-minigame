## Location
`backends/backend-minigame/src/main/java/com/github/xpenatan/gdx/teavm/backends/minigame/MiniGameNet.java` — `sendHttpRequest()` method

## The Bug
MiniGameNet uses `org.teavm.jso.ajax.XMLHttpRequest` — TeaVM's wrapper around the browser's XMLHttpRequest API. WeChat Mini Game **does not support XMLHttpRequest**. It uses `wx.request()` for HTTP calls.

This means **ALL network requests will fail** — no HTTP GET/POST, no asset downloads from URLs, no API calls.

## Fix
Replace XMLHttpRequest with `wx.request()`:
1. Add WX.request() binding in WX.java:
```java
@JSBody(params = {"options", "successCb", "failCb"}, script = 
    "options.success = function(res) { successCb.invoke(res.data, res.statusCode, res.header); };" +
    "options.fail = function(err) { failCb.invoke(err.errMsg); };" +
    "wx.request(options);")
public static native void request(JSObject options, SuccessCallback successCb, FailCallback failCb);
```

2. Rewrite sendHttpRequest() to use WX.request() with proper method/headers/body handling

## Severity: High — any game that uses Gdx.net.sendHttpRequest() will crash. Low for games that don't make HTTP calls.

## Note
This may not be a priority for a card game (Solitaire) that loads assets from local storage, but it's needed for any game with online features, leaderboards, asset downloads, etc.
