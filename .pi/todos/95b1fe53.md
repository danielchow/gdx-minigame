## Location
`backends/backend-minigame/src/main/java/com/github/xpenatan/gdx/teavm/backends/minigame/MiniGameInput.java` line ~79-80

## The Bug
```java
int x = (int) touch.getClientX();
int y = (int) touch.getClientY();
```

On WeChat Mini Game, touch.clientX/clientY are in **CSS pixels** (logical pixels). But if `MiniGameApplicationConfiguration.usePhysicalPixels` is true, the canvas dimensions are set to physical pixels (CSS pixels × devicePixelRatio).

Example: iPhone with pixelRatio=3
- Screen: 390×844 CSS pixels  
- Canvas: 1170×2532 physical pixels (if usePhysicalPixels=true)
- Touch at center: clientX=390, clientY=422 (CSS pixels)
- GL viewport expects: 585, 1266 (center of 1170×2532)

The touch coordinates are at 1/3 of the canvas size — input is wildly off.

## How WebInput Handles This
WebInput uses `getRelativeX()` which accounts for canvas offset AND CSS scaling:
```java
float xScaleRatio = target.getWidth() * 1f / getClientWidth(target);
return Math.round(xScaleRatio * (e.getClientX() - ...));
```

## Fix
In `handleTouchEvent()`, apply devicePixelRatio scaling when `usePhysicalPixels` is true:
```java
double density = config.usePhysicalPixels ? WX.getPixelRatio() : 1.0;
int x = (int)(touch.getClientX() * density);
int y = (int)(touch.getClientY() * density);
```

Note: MiniGameInput needs access to the config to check `usePhysicalPixels`. It currently takes `MiniGameApplication` and `HTMLCanvasElement` in the constructor. Could pass the config or just the boolean flag.

## Severity: High for games using usePhysicalPixels=true; Medium otherwise (CSS pixel coordinates may work fine if canvas is also in CSS pixels)
